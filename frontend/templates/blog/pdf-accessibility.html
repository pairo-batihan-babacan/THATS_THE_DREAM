PDF Accessibility Guide: Making Documents Inclusive

March 16, 20247 min read

Accessible PDFs can be read by everyone — including people who use screen readers, have low vision, are colour blind, or navigate using a keyboard rather than a mouse. Creating accessible PDFs isn't just about legal compliance (though it is required for public sector organisations in many countries) — it's about ensuring your documents can actually communicate their content to all readers. This guide explains what makes a PDF accessible and how to achieve it.

Why PDF Accessibility Matters

According to the World Health Organization, over 2.2 billion people worldwide have a vision impairment. Millions more have motor disabilities that prevent them from using a mouse, or cognitive differences that make dense or poorly structured documents harder to process. An inaccessible PDF excludes these users entirely.

From a legal standpoint:

  • United States: ADA (Americans with Disabilities Act) and Section 508 of the Rehabilitation Act require accessible digital documents for federal agencies and many public-facing organisations.
  • European Union: The Web Accessibility Directive (2016/2102) requires public sector bodies to ensure documents meet WCAG 2.1 Level AA.
  • United Kingdom: The Equality Act 2010 and the Public Sector Bodies Accessibility Regulations 2018 impose similar requirements.

The Foundation: Tagged PDFs

The single most important accessibility feature of a PDF is whether it is tagged. Tags are hidden markers within the PDF structure that define the reading order and semantic meaning of content: "this is a heading level 1", "this is a paragraph", "this is a table", "this image has this alt text."

Screen readers use these tags to navigate and read the document. An untagged PDF is essentially a visual image to a screen reader — it either reads nothing or reads content in a garbled, meaningless order.

Tagged PDFs are created by exporting from Microsoft Word, Adobe InDesign, or Adobe Acrobat (Pro) with accessibility options enabled. Simply scanning a document creates an untagged image-only PDF — completely inaccessible to screen readers without additional processing.

Key Accessibility Requirements

1. Document Language

Set the document language in the PDF metadata. Screen readers use this to select the correct pronunciation rules and voice. A PDF marked as English will be read with English pronunciation; a French PDF without a language setting might be read with incorrect English pronunciation, making it incomprehensible.

2. Heading Structure

Use proper heading levels (H1, H2, H3) consistently. Don't make text look like a heading by just making it larger and bold — apply the proper heading tag. Screen reader users navigate long documents by jumping between headings, so a clear, logical hierarchy is essential for usability. The rule: one H1 per document (the title), H2 for major sections, H3 for subsections.

3. Alternative Text for Images

Every meaningful image in an accessible PDF must have descriptive alt text explaining what the image shows. For a chart, the alt text should describe the data being illustrated, not just say "chart." Decorative images (visual dividers, background patterns) should be marked as "decorative" or "artifact" so screen readers skip them rather than reading "image" uselessly.

4. Colour Contrast

Text must have sufficient contrast against its background colour. WCAG 2.1 Level AA requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text (18pt or 14pt bold). This affects readers with low vision or colour blindness. Never convey information through colour alone — for example, don't say "items in red are urgent" if that's the only indicator.

5. Reading Order

The reading order of tagged content must match the logical visual order. In a multi-column document, the tags should flow left-to-right within each column before moving to the next — not zigzag across columns. PDFs with complex layouts (magazine-style multi-column, sidebars, call-out boxes) often have reading order problems that need manual correction in Acrobat Pro's Order panel.

6. Tables

Data tables must have proper header cells marked as TH (table header) rather than TD (table data). Each data cell should be programmatically associated with its header cells so screen reader users know which column and row they're in. Avoid using tables for purely visual layout purposes — they confuse screen readers.

7. Form Fields

Interactive form fields must have descriptive labels. A text field without a label is incomprehensible to a screen reader — the user is told "text input" but has no idea what to type. Tab order through form fields must be logical (typically top-to-bottom, left-to-right).

8. Hyperlinks

Link text must be descriptive. "Click here" or "read more" tells a screen reader user nothing about the destination. Use descriptive text: "Download the 2024 Annual Report (PDF)" or "Visit the W3C Accessibility Guidelines website."

9. Document Title

Set a meaningful document title in the PDF properties. Screen readers announce this title when the document opens. "Untitled" or "document1" is unhelpful — use the full document title.

How to Create an Accessible PDF

The best approach is to build accessibility into the source document before converting to PDF:

  1. In Microsoft Word, use proper heading styles (not manual formatting), add alt text to images (right-click → Format Picture → Alt Text), ensure table headers are marked, and use descriptive hyperlinks.
  2. Export to PDF using Word's built-in PDF export (File → Save As PDF) with "Document structure tags for accessibility" checked.
  3. In Adobe Acrobat Pro, run the Accessibility Checker (Tools → Accessibility → Full Check) and fix any reported issues.
  4. Check reading order in the Order panel and fix any logical sequence problems.
  5. Test with a screen reader (NVDA on Windows is free) by navigating with the keyboard only.

Testing PDF Accessibility

  • Adobe Acrobat Pro — Built-in Accessibility Checker identifies many common issues automatically.
  • PAC 2021 (PDF Accessibility Checker) — Free tool specifically for checking PDF/UA compliance.
  • NVDA — Free screen reader for Windows. Test your document by navigating it with NVDA enabled and your eyes closed.
  • VoiceOver (macOS/iOS) / TalkBack (Android) — Built-in screen readers for Apple and Google devices.

Accessible PDFs Start with Accessible Content

The most efficient way to create accessible PDFs is to prioritise accessibility when creating the source document in Word, InDesign, or Google Docs. Retrofitting accessibility into an existing untagged PDF is possible but much more labour-intensive. Build good habits from the start: use heading styles, write descriptive alt text, and maintain logical structure.
